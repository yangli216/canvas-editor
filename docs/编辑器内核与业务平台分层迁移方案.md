# 编辑器内核与业务平台分层迁移方案

更新日期：2026-05-26

## 目标

将当前仓库从“编辑器能力 + 模板平台业务 + 演示应用”混合结构，演进为边界清晰、可逐步拆包的分层结构。

分层目标不是立即拆成多个仓库，而是先在同一工作区内建立稳定边界：

- 编辑器内核只承载通用编辑、渲染、命令、插件和模板编译能力。
- 病历模板/病历运行平台承载模板管理、病历留痕、发布准入、权限、质控等业务能力。
- 演示应用与未来工作台应用通过平台层接入内核，而不是直接在入口文件里拼装业务逻辑。

## 当前问题

当前结构已经出现典型混层现象：

- `src/editor` 同时承担内核与部分业务模板运行职责。
- `src/components/template-designer` 已经包含大量平台级工作台能力，而非单纯编辑器组件。
- [src/main.ts](src/main.ts) 既负责 demo 应用启动，也直接编排模板工作台、病历实例、留痕面板等业务流程。

继续沿现状迭代会带来四类问题：

- 内核与业务平台发布节奏冲突。
- 业务模型持续反向污染编辑器内核。
- 单次改动回归面过大。
- 后续接入权限、质控、审计、真实接口后，入口文件和组件层会迅速失控。

## 目标结构

建议逐步收敛到以下目录心智：

```text
src/
├── editor/                      # 编辑器内核与可复用模板基础能力
├── platform/                    # 病历模板平台/病历运行平台业务层
│   └── emr-workbench/           # 模板工作台、病历留痕、发布准入等编排入口
├── apps/                        # 演示应用与未来平台应用入口
│   └── editor-demo/             # 当前 demo 页面启动与装配
├── components/                  # 可复用 UI 组件
└── utils/                       # 通用工具
```

其中职责边界应为：

- `editor/`：不直接感知科室、角色、发布状态、审核流、权限策略等医院业务概念。
- `platform/`：组合模板注册、病历实例、权限、质控、发布等领域能力，对外提供业务工作台入口。
- `apps/`：只负责装配 UI 外壳、路由、应用级状态和环境配置。

## 已完成第一步

本轮已经落下一步真实拆分：

- 新增 `src/platform/emr-workbench` 作为平台层起点。
- 将 `main.ts` 中模板工作台、病历实例 store、病历留痕面板、模板运行绑定等业务装配逻辑抽到平台层 `bootstrapEmrWorkbench()`。
- `main.ts` 现在只消费平台层公开入口，不再直接编排模板工作台细节。

在此基础上又继续完成第二步起手：

- 新增 `src/apps/editor-demo` 承接 demo 应用启动与编辑器初始化装配。
- `main.ts` 改为消费 demo 应用层 `bootstrapEditorDemoApp()`，不再直接初始化 demo 数据、全局实例和平台接线。
- 新增 `platform/emr-workbench/domain/`，将模板发布准入、试运行、审计日志、病历实例追溯、病历迁移等领域动作收口为 service。
- 新增 `platform/emr-workbench/modules/permission-center`、`quality-center`、`audit-center` 作为后续独立子模块边界。

这一步的意义是：

- 后续新增模板权限、病历质控、运营台、审计台时，有明确的代码落点。
- 未来要拆分成多包工作区或独立业务应用时，不需要从 `main.ts` 重新识别边界。

## 后续迁移顺序

### 第一阶段：继续把业务编排移出入口文件

- 将模板管理工作台入口、病历留痕面板入口、业务字段中心入口、规则中心入口继续从 `main.ts` 抽到 `platform/`。
- 把演示环境专属的 DOM 查询和按钮挂载收敛到 `apps/` 目录，避免 `platform/` 直接依赖 demo 页面结构。
- 继续把主菜单、工具栏、调试辅助和 demo 交互装配从 `main.ts` 下沉到 `apps/editor-demo/`。

### 第二阶段：收敛领域层

- 将模板注册、发布准入、病历实例、病历迁移、留痕、权限预留、质控规则等归并为平台领域模块。
- `editor/` 只保留模板 DSL、编译、运行时取值、基础适配器抽象等通用能力。
- 继续把 `TemplateManager` 里的 registry/store 直连调用替换为 `platform/emr-workbench/domain/` service。

### 第三阶段：引入应用层

- 已建立 `src/apps/editor-demo/`，后续继续把当前 demo 启动逻辑迁入应用层。
- 如果模板平台 UI 继续扩大，可新增 `src/apps/emr-workbench/` 作为真正的平台应用入口。

### 第四阶段：评估升级为多包工作区

当以下条件成立时，再考虑升级为 monorepo 多包结构：

- 编辑器内核需要独立版本演进。
- 病历模板平台有独立发布周期。
- 权限、质控、审计开始接入真实后端和组织模型。
- 团队开始出现“内核开发”和“平台开发”的分工。

## 迁移原则

- 先拆边界，再拆仓库。
- 先抽装配层，再抽领域层，最后再考虑多包化。
- 任何一次拆分都必须保持现有运行链路和测试可通过。
- 新增业务能力优先落到 `platform/`，避免继续把平台逻辑堆进 `main.ts` 或 `editor/`。
