## 产品愿景

Canvas Editor 的模板方向已经从“编辑单页病历样式”推进到“面向医院病历文书的可视化模板平台”。下一阶段不再继续堆叠单点编辑能力，而是把已形成的模板设计、发布准入、试运行验证、临床使用、回滚修复、审计追溯和平台模块化能力组织成面向病历质量管理与归档运营的临床工作台。

目标是让平台从“模板能上线并可回滚”升级为“病历能被持续质控、缺陷能被闭环整改、归档能被冻结追溯、模板问题能反向修复”。

## 当前里程碑：P2 病历质控与归档运营闭环

### 产品定位

P2 的重心从泛化的“模板平台治理”前移到临床侧的“病历质控与归档闭环”。平台需要围绕医生书写、科室质控、病案归档、问题整改和归档后追溯，形成一条可执行的病历质量管理链路：病历创建后能被质控规则检查，问题能被定位到字段和模板版本，整改能回到医生或模板维护人员，归档前能完成必要校验，归档后能保留快照、留痕和回溯能力。

P2 不追求一次性实现真实病案室系统、电子签章或医院归档接口，而是先在现有前端平台和 mock/domain service 基础上，把病历质控、缺陷整改、归档检查、归档快照和模板问题反哺产品化，为后续接入真实病案首页、质控规则库、归档服务和权限系统预留边界。

### 核心原则

- 质控前置：病历书写中、提交前、归档前都应能触发对应强度的质控检查。
- 缺陷可定位：每个质控问题必须尽量定位到模板、版本、病历实例、字段路径或书写事件。
- 整改可追责：缺陷从发现、分派、整改、复核到关闭要有状态流转和操作留痕。
- 归档可冻结：归档病历需要保存模板快照、字段值、结构化导出、质控结论和归档时间线。
- 问题可反哺：高频病历缺陷要能反向暴露模板设计、字段绑定或规则配置问题。
- 架构可扩展：病历质控与归档能力下沉到 `src/platform/emr-workbench/`，编辑器内核保持通用。

### 暂不纳入 P2

- 暂不接入真实医院账号权限系统，仍以角色模型和前端 domain service 验证交互闭环。
- 暂不接入真实 HIS/EHR、病案首页或归档服务网络接口，继续使用 adapter 抽象和 mock 数据验证链路。
- 暂不实现真实电子签章、CA 认证、病案室审批流或医保编码上传。
- 暂不做跨医院多租户商业后台，先完成单院区、多科室的质控与归档模型。
- 暂不做复杂 BI 大屏，只保留病历质控和归档运营所需的轻量统计和待办视图。
- 暂不做 AI 自动生成病历或模板，避免把质控归档里程碑扩散为内容生成里程碑。

## 后续执行建议

优先从“病历质控工作台”切入。它能复用现有病历实例、模板版本快照、书写留痕、发布准入和结构化导出能力，先把待质控、待整改、待归档病历集中展示出来。

第二优先级建议推进“病历归档与归档质检”。它能把 P1 已完成的病历快照、字段值、结构化导出和留痕时间线进一步冻结为归档资产，是从模板平台走向真实病历管理场景的关键一步。
