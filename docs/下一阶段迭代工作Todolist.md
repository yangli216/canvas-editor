## 下一里程碑节点：P1 模板上线与临床运行闭环

### 产品定位

当前模板平台已经具备模板设计、模板管理、预览测试、版本发布、业务字段绑定、HIS/EHR mock 数据回填、结构化导出和病历实例存储分析等基础能力。下一里程碑不再继续堆叠单点编辑功能，而是把能力组织成医院真实可落地的“上线闭环”：模板从设计草稿进入发布准入、试运行验证、临床使用、问题回滚和运营治理。

目标是让模板中心从“能管理模板”升级为“能支撑模板上线运营”，让设计器从“能配置文书”升级为“能交付可运行、可追溯、可回滚的院内模板资产”。

### 核心原则

- 模板资产必须可归属：每个模板需要明确科室、文书类型、负责人、适用场景和上线状态。
- 上线前必须可验证：发布前能看到结构、规则、数据绑定、必填项、页眉页脚和迁移风险的统一准入结果。
- 临床使用必须可追溯：病历实例必须绑定模板版本快照，避免模板后续变化影响历史病历回显。
- 问题处理必须可回退：线上模板出现问题时，应能快速定位影响范围、回滚版本或复制修订版本。
- 运维治理必须可持续：模板中心要支持按科室、状态、风险、数据源和最近使用情况持续运营。

### 里程碑验收口径

- [ ] 模板维护人员能从工作台按科室/文书类型/状态快速找到目标模板。
- [ ] 模板发布前能看到统一准入报告，阻断项会阻止发布并能定位修复。
- [ ] 模板能通过试运行验证保存回归样本，并在发布流程中展示验证状态。
- [ ] 使用已发布模板创建病历时会保存模板版本快照，历史病历回显不受模板修改影响。
- [x] 病历实例关键动作可在病历书写页中查看留痕时间线，并能看到字段级变更摘要。
- [x] 模板工作台仅保留“引用病历追溯”，用于查看模板版本被哪些病历引用及最近实例轨迹。
- [ ] 线上模板修复能走“复制为修订草稿 -> 验证 -> 发布 -> 回滚/替换”的完整路径。
- [ ] 数据绑定覆盖率、缺失字段和结构化导出结果能被维护人员直接查看和排查。
- [ ] 关键模板操作有日志、有确认、有影响范围提示。

### 暂不纳入本里程碑

- 暂不接入真实医院后端账号权限系统，仅保留角色与权限模型接口。
- 暂不实现真实 HIS/EHR 网络接口，只继续使用 adapter 抽象和 mock adapter 验证链路。
- 暂不做跨医院多租户后台，先以单院区/多科室运营模型打通产品闭环。
- 暂不追求复杂 BI 看板，优先完成模板上线所需的准入、验证、使用、回滚和审计。
