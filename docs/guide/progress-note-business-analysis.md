# 病程记录业务支撑分析

本文档聚焦“病程记录”这一类医疗文书，目标不是讨论通用模板能力，而是判断当前模板平台对病程业务的支撑程度，并给出面向产品化落地的推荐迭代顺序。

## 一、结论摘要

当前系统对病程记录的支撑程度可定义为：

1. 通用病程模板录入可用。
2. 模板版本与历史病历保护方向正确。
3. 病程业务规则、时效、审签和连续记录模型尚未闭环。

一句话概括：

> 当前已经具备“病程模板引擎”，但还没有形成“病程业务系统”。

## 二、病程书写的核心业务要求

对于住院病程记录，产品上至少需要覆盖以下能力：

1. 记录类型差异化：首次病程、日常病程、上级医师查房、交接班、疑难病例讨论、术后病程等不应只靠一个下拉选项区分。
2. 固定骨架要求：病情变化、客观情况、检验检查、诊疗分析、处理措施、后续计划、记录者与签名等信息要可约束。
3. 时效要求：记录是否按时完成、是否补记、是否超时，需有明确规则。
4. 角色责任：住院医师、主治医师、主任查房记录要求不同，部分场景有上级确认责任。
5. 历史稳定性：模板升级不能破坏既往病程的回显、审计与签名留痕。

## 三、当前已支撑的能力

### 1. 病程模板骨架

当前已有通用病程模板示例：

- [src/editor/template/examples/progressNote.ts](../../src/editor/template/examples/progressNote.ts)

并且本轮继续补充了病程模板族：

1. 通用病程记录模板。
2. 首次病程记录模板。
3. 日常病程记录模板。
4. 上级医师查房记录模板。

这意味着系统已经从“单一病程模板”升级为“病程模板族起点”。

### 2. 结构化运行时

当前运行时支持：

1. 平铺值导出。
2. 结构化值导出。
3. 按 section / group / businessCode / permission / dataSource 等维度分桶。

参考：

- [src/editor/template/TemplateRuntime.ts](../../src/editor/template/TemplateRuntime.ts)

这对病程后续对接 HIS、病历首页、质控和统计都很重要。

### 3. 病历快照与迁移

当前病历存储层已经支持：

1. 模板快照绑定。
2. 迁移预览。
3. 迁移记录。
4. autosave 与正式持久化 DTO。

参考：

- [src/editor/template/TemplateDocumentStore.ts](../../src/editor/template/TemplateDocumentStore.ts)
- [docs/guide/emr-storage-implementation.md](./emr-storage-implementation.md)

这部分对病程文书尤其关键，因为病程是高频记录、长期追溯、模板持续演进的典型场景。

## 四、当前仅部分支撑的能力

### 1. 规则引擎

当前规则引擎实际稳定支持的是：

1. 显隐。
2. 只读。
3. 隐藏。
4. 级联。

参考：

- [src/editor/template/TemplateRuleEngine.ts](../../src/editor/template/TemplateRuleEngine.ts)

这说明系统已经具备病程规则化的基础，但还没有落到病程专属业务规则上。

### 2. 业务元数据

当前字段元数据已经支持：

1. `businessCode`
2. `exportPath`
3. `permission`
4. `dataSource`
5. `listeners`
6. `tags`

参考：

- [src/editor/template/index.ts](../../src/editor/template/index.ts)

这意味着未来做病程分类、病程质控、病程权限和业务提取是有扩展面的，但目前模板示例里还没有系统化地使用到病程业务元数据模型。

## 五、当前明显不足的能力

### 1. 病程时效规则

当前尚未看到真正落地的：

1. 入院后首次病程时点约束。
2. 日常病程记录频次约束。
3. 病危病重变化后的限时记录约束。
4. 超时未写提醒与质控拦截。

### 2. 审签链路

当前已经有 `signed` 状态，但还没有形成病程业务上的完整审签闭环，例如：

1. 上级医师查房记录的签认。
2. 签名前后可编辑边界。
3. 补记与签名留痕。

### 3. 连续病程模型

病程业务通常不是“单份文书”，而是同一住院过程下的连续记录集合。当前系统更偏“单文书实例”，还没有把病程条目流、时间线和排序模型抽出来。

### 4. 病程质控专属校验

当前尚未固化为产品能力的典型病程质控包括：

1. 首次病程缺病例特点。
2. 首次病程缺诊断依据。
3. 日常病程缺病情变化或处理计划。
4. 上级查房缺上级意见。
5. 签名后仍允许修改且无修订留痕。

## 六、本轮实现推进

本轮实现的核心动作是：

1. 将原单一病程模板升级为病程模板族。
2. 把“首次病程 / 日常病程 / 上级医师查房”从 `noteType` 下拉中抽成独立模板入口。
3. 为后续病程规则、时效、审签和模板迁移做业务承载面准备。

这属于产品化推进中的第一步：

> 先把“模板类型”拆开，再把“业务规则”加上，最后再做“流程与时效”。

## 七、推荐的下一步实现顺序

### 阶段 1：病程模板族化

本轮已启动：

1. 通用病程记录。
2. 首次病程记录。
3. 日常病程记录。
4. 上级医师查房记录。

### 阶段 2：病程专属规则

下一步建议优先补：

1. 记录类型专属必填规则。
2. 上级查房意见必填。
3. 签名后只读。
4. 已归档禁止直接迁移。

### 阶段 3：病程连续记录模型

建议新增“住院过程下的病程条目集合”模型：

1. 一个 encounter 对应多个 progress entry。
2. 每个 entry 绑定模板版本与快照。
3. 支持按时间线浏览、补记、修订和迁移预览。

### 阶段 4：病程质控中心

把病程业务真正收口为产品能力：

1. 超时未写。
2. 必填缺失。
3. 上级意见缺失。
4. 修订留痕缺失。
5. 已签名违规修改。

## 八、最终判断

当前系统对病程业务的支撑性判断如下：

1. 模板与存储底座：较强。
2. 通用病程录入能力：可用。
3. 病程专属业务规则：部分支撑。
4. 病程审签、时效、质控：明显不足。

因此更准确的表述应是：

> 这套系统已经能承载病程记录模板化和结构化存储，但距离医院级病程业务闭环仍需要继续补规则、流程和质控能力。
