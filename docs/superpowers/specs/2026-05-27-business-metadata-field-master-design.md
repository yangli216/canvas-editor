# 字段主数据管理台设计稿

日期：2026-05-27

## 1. 背景

当前仓库中的“业务字段中心”已经具备以下能力：

- 基于模板 schema 扫描字段资产，形成字段清单
- 按关键字、分组、数据源、权限标签筛选字段
- 手工维护字段元数据并回写模板 schema
- 通过业务字段快配批量回写控件类型、业务编码、数据源和导出路径
- 在模板设计器属性栏内复用字段快配与推荐逻辑

但现状的核心问题是：

- 业务字段中心仍然以“模板字段资产聚合”为主，而不是独立的“字段主数据管理台”
- 业务元数据主记录不存在，导致多个模板之间容易出现重复 businessCode、数据源不一致、权限标签漂移等问题
- 模板字段与业务元数据之间缺少稳定的引用关系，现阶段更多是“把元数据复制进字段”
- 无法从字段主数据视角反查模板使用范围，也无法从模板字段视角明确看到自己绑定的是哪条主数据记录

因此需要把“业务字段中心”演进为“字段主数据管理台”，将业务元数据治理从模板聚合视角升级为主数据主源视角，同时支持模板字段与业务元数据的双向关联。

## 2. 目标

本期目标：

1. 建立字段主数据实体，作为业务元数据唯一主源。
2. 建立模板字段与主数据字段之间的显式绑定关系，支持双向查询。
3. 将当前业务字段中心升级为字段主数据管理台，支持主数据维护、引用查看和风险治理。
4. 在模板设计器中将核心业务元数据编辑方式收口为“绑定主数据字段”，而不是继续自由手填。
5. 保证旧模板兼容可运行，并支持从现有模板字段资产反向初始化主数据候选。

## 3. 非目标

本期明确不做以下内容：

- 不把字段规则中心、默认值策略中心、字典主数据中心一起并入首期范围
- 不实现主数据台与模板设计器的双主源同步编辑
- 不接入真实 HIS/EHR 主数据接口，继续使用现有 domain service 与 mock 数据模式验证交互闭环
- 不要求一次性迁移所有历史模板到主数据绑定模式
- 不在编辑器内核中承载完整主数据治理逻辑

## 4. 方案选择

### 4.1 候选方案

#### 方案 A：模板聚合型

以模板字段元数据为主源，业务字段中心继续做聚合、去重、筛选和回写。

优点：

- 改动最小
- 可复用现有字段资产清单能力

缺点：

- 无法形成真正的主数据台
- 模板间冲突无法被稳定治理
- 双向关联仍然停留在“扫描结果”，不是“引用关系”

#### 方案 B：主数据主源型

新增字段主数据实体，模板字段只持有主数据引用和兼容元数据。

优点：

- 符合“字段主数据管理台”的定位
- 双向关联边界清晰
- 后续可扩展到规则、字典、血缘和影响评估

缺点：

- 需要新增实体、服务和迁移兼容层

#### 方案 C：双主源同步型

主数据台和模板设计器都可以修改核心业务元数据，并做自动同步。

优点：

- 表面上最灵活

缺点：

- 冲突处理复杂
- 审计、优先级、回滚和状态治理成本过高
- 首期风险明显偏大

### 4.2 结论

选择方案 B：主数据主源型。

原因：

- 该方案最符合“字段主数据管理台”的目标定义
- 能以最清晰的边界支持“模板字段和业务元数据双向关联”
- 可以通过兼容层平滑承接已有模板字段元数据，不需要一次性重写现有模板链路

## 5. 现有能力与改造方向

### 5.1 当前代码基线

当前相关能力主要分布在以下位置：

- `src/editor/template/index.ts`
  - `ITemplateFieldMetadata` 已承载 `businessCode`、`group`、`exportPath`、`permission`、`dataSource`、`listeners`、`tags`
- `src/editor/template/TemplateRuntime.ts`
  - 已基于模板字段元数据建立运行时索引和结构化导出能力
- `src/editor/template/TemplateFieldQuickPreset.ts`
  - 已定义业务字段快配和推荐逻辑
- `src/platform/emr-workbench/modules/business-field-center/service.ts`
  - 已从模板中扫描字段资产并构建字段中心视图模型
- `src/components/template-designer/TemplateManager.ts`
  - 已提供业务字段中心弹层、字段快配、字段资产维护等入口

### 5.2 本期改造原则

1. 编辑器模板结构只新增最小必要引用字段，不把平台治理逻辑侵入内核。
2. 主数据治理逻辑优先落到 `src/platform/emr-workbench/`。
3. 运行时读取业务元数据时应优先使用主数据实体，旧字段元数据作为兼容回退。
4. 现有字段快配能力要保留，但其目标从“直接写模板字段元数据”演进为“生成或绑定主数据候选”。

## 6. 领域模型设计

### 6.1 字段主数据实体

新增 `BusinessMetadataField`：

```ts
interface IBusinessMetadataField {
  id: string
  code: string
  name: string
  group: string
  dataSource: string
  permission: string
  exportPath: string
  tags?: string[]
  status: 'active' | 'inactive'
  description?: string
  createdAt: number
  updatedAt: number
}
```

字段说明：

- `id`：主数据记录主键，供模板字段绑定使用
- `code`：业务编码，作为治理和去重的核心键
- `name`：主数据展示名称
- `group`：业务分组，如基本信息、就诊信息、病程记录
- `dataSource`：业务来源，如 `his.patient`、`his.visit`、`emr.notes`
- `permission`：业务权限标签
- `exportPath`：结构化导出路径
- `tags`：业务标签，用于筛选和运营标记
- `status`：主数据启停状态
- `description`：可选描述，用于说明字段业务语义

### 6.2 模板字段绑定实体

新增 `TemplateFieldBinding`：

```ts
interface ITemplateFieldBinding {
  templateId: string
  fieldId: string
  metadataFieldId: string
  bindMode: 'reference'
  boundAt: number
}
```

设计说明：

- 首期只支持 `reference` 绑定模式，不支持复制模式和覆盖模式
- 模板字段绑定主数据后，核心业务元数据由主记录提供
- 双向关联查询本质上基于这张绑定关系完成

### 6.3 模板字段兼容层

在 `ITemplateFieldMetadata` 上新增兼容字段：

```ts
interface ITemplateFieldMetadata {
  metadataFieldId?: string
  businessCode?: string
  group?: string
  exportPath?: string
  permission?: string
  dataSource?: string
  listeners?: string[]
  tags?: string[]
}
```

兼容规则：

1. 若 `metadataFieldId` 存在且能找到主数据记录，则以主数据记录为准。
2. 若 `metadataFieldId` 不存在，则继续读取旧元数据字段。
3. 若 `metadataFieldId` 存在但主数据记录缺失，则标记为“失效绑定”，并在管理台中暴露风险。

这样可以保证历史模板不需要立刻迁移，也能为新模板提供标准主数据绑定能力。

## 7. 服务与模块设计

### 7.1 新增领域服务

建议在平台层新增字段主数据领域服务，位置建议为：

- `src/platform/emr-workbench/domain/business-metadata/`

首期服务职责：

- 管理主数据字段的新增、编辑、停用、查询
- 管理模板字段绑定/解绑
- 提供按主数据字段反查模板字段引用能力
- 提供从模板字段资产反向生成主数据候选能力
- 提供冲突检测能力，如重复 `businessCode`、同码不同源、同码不同权限

建议暴露如下 API：

- `listFields()`
- `getField(id)`
- `createField(input)`
- `updateField(id, patch)`
- `deactivateField(id)`
- `listBindings()`
- `bindTemplateField({ templateId, fieldId, metadataFieldId })`
- `unbindTemplateField({ templateId, fieldId })`
- `getFieldUsage(metadataFieldId)`
- `buildFieldCandidatesFromTemplates(items)`
- `detectFieldConflicts(items)`

### 7.2 现有业务字段中心模块的演进

现有模块：

- `src/platform/emr-workbench/modules/business-field-center/`

改造方向：

- 从“字段资产汇总中心”升级为“字段主数据管理台”
- 视图模型的核心输入从“模板扫描结果”扩展为“主数据字段 + 绑定关系 + 模板候选字段”
- 原有字段快配、字段维护入口改造为“主数据编辑”和“绑定管理”入口

建议保留目录名不变以降低改动范围，但对外文案与职责升级为“字段主数据管理台”。

## 8. UI 设计

### 8.1 首期信息架构

首期管理台分为三块：

#### A. 主数据列表区

展示所有主数据字段，支持：

- 关键字筛选：编码、名称、描述
- 维度筛选：分组、数据源、权限、状态
- 统计摘要：主数据总数、启用数、已绑定模板字段数、未绑定候选字段数、冲突项数
- 操作：新增、编辑、停用、查看详情

#### B. 主数据详情抽屉

展示单条主数据字段详情：

- 基本信息
- 业务标签和描述
- 被哪些模板字段引用
- 最近变更摘要
- 冲突提示或失效绑定提示

#### C. 关联视图区

围绕某条主数据字段展示：

- 已绑定模板字段
- 推荐绑定候选字段
- 未绑定但疑似同义字段
- 同码冲突或元数据漂移风险

### 8.2 模板设计器侧改造

当前模板设计器支持直接编辑业务编码、字段分组、数据源、权限标签和导出路径。

本期调整为：

- 保留业务元数据分区，但核心字段改为“主数据绑定”模式
- 新增“搜索并绑定主数据字段”入口
- 绑定后展示继承信息摘要：编码、分组、数据源、权限、导出路径
- 新增“解除绑定”动作
- 未绑定字段继续显示兼容元数据，但视觉上标记为“未纳管”

这样可以让模板维护人员逐步从自由输入迁移到主数据绑定。

### 8.3 字段快配的调整

当前字段快配是直接对模板字段打补丁。

本期建议调整为两条路径：

1. 若存在命中的主数据字段，则优先推荐直接绑定。
2. 若不存在命中的主数据字段，则允许基于快配生成主数据候选，再绑定到模板字段。

这样既保留现有效率能力，又不破坏“主数据唯一主源”的边界。

## 9. 数据流设计

### 9.1 主数据到模板字段

1. 管理员在字段主数据台创建或维护主数据字段。
2. 模板维护人员在设计器或管理台选择模板字段。
3. 系统通过搜索、推荐或候选匹配找到主数据字段。
4. 建立 `TemplateFieldBinding`，同时在模板字段元数据里写入 `metadataFieldId`。
5. 模板运行时读取字段业务元数据时优先解析主数据字段。

### 9.2 模板字段到主数据台

1. 管理台扫描现有模板字段。
2. 对已绑定字段，汇总为主数据字段的使用清单。
3. 对未绑定字段，生成候选池并按 businessCode / label / placeholder / dataSource 做聚合。
4. 将冲突、重复、未纳管字段暴露给管理员处理。

### 9.3 兼容运行时解析

运行时读取字段元数据时采用以下顺序：

1. 读取 `metadataFieldId`
2. 若可命中主数据，则使用主数据字段的核心属性
3. 追加模板字段自身保留的非主数据属性，如 `listeners`、模板局部标签等
4. 若未命中主数据，则回退到旧模板字段元数据

该规则既保证主数据主源，又兼容历史模板。

## 10. 冲突与错误处理

### 10.1 首期需要识别的风险

- 重复 `businessCode`
- 同一 `businessCode` 对应不同 `dataSource`
- 同一 `businessCode` 对应不同 `permission`
- 模板字段已绑定，但主数据记录已不存在
- 模板字段未绑定且核心元数据缺失

### 10.2 处理策略

- 管理台通过风险列表集中展示冲突项
- 对重复编码但元数据完全一致的候选项，可允许人工合并为同一主数据字段
- 对同码不同源、同码不同权限的情况，必须人工确认，不做自动合并
- 对失效绑定显示警告并允许重新绑定或解除绑定

## 11. 测试设计

首期自动化测试至少覆盖以下场景：

1. 主数据字段新增、编辑、停用和查询
2. 模板字段绑定主数据、解绑主数据
3. 绑定后从主数据字段反查模板引用
4. 运行时优先读取主数据记录，旧模板走兼容回退
5. 从模板字段资产构建主数据候选
6. 重复编码与元数据冲突检测
7. 字段主数据管理台 view model 的统计、筛选和风险列表
8. 模板设计器属性面板在已绑定、未绑定、失效绑定三种状态下的展示差异

建议测试分布：

- `tests/platform/`：字段主数据台 view model 与 platform service 测试
- `tests/template/`：模板运行时兼容解析与绑定行为测试
- `tests/editor/` 或 `tests/components/`：属性面板交互和展示测试

## 12. 实施分阶段建议

### 阶段 1：主数据模型与服务底座

- 新增主数据实体和绑定实体
- 新增领域服务和冲突检测
- 提供从模板字段资产反向生成主数据候选能力

### 阶段 2：字段主数据管理台 UI

- 将业务字段中心 UI 升级为主数据台
- 接入主数据列表、详情抽屉、关联视图和风险提示

### 阶段 3：模板设计器绑定改造

- 属性面板改为主数据绑定模式
- 保留兼容展示与解除绑定能力
- 将快配入口改造为绑定优先、候选生成次之

### 阶段 4：运行时兼容与收口

- 模板运行时优先解析主数据字段
- 完善旧模板兼容逻辑
- 补齐测试与迁移提示

## 13. 验收口径

当满足以下条件时，认为首期达成：

- 平台层存在独立的字段主数据实体与绑定关系模型
- 字段主数据管理台可维护主数据字段，而不是只能扫描模板字段
- 任一主数据字段都可反查被哪些模板字段引用
- 模板字段可以绑定或解绑主数据字段
- 绑定后的模板字段不再依赖手工维护核心业务元数据
- 历史模板在未绑定主数据的情况下仍可正常运行
- 冲突项和失效绑定可被集中识别和处理

## 14. 风险与后续扩展

### 14.1 风险

- 旧模板元数据质量不一致，初次生成主数据候选时可能噪音较大
- 若模板设计器仍允许长期自由编辑核心元数据，会削弱主数据主源的落地效果
- 若运行时解析路径处理不清，可能出现主数据和兼容元数据读值不一致

### 14.2 后续扩展方向

- 主数据字段级规则与默认值策略
- 业务字典主数据联动
- 按模板版本和归档病历评估字段主数据变更影响范围
- 主数据审计日志与审批流

## 15. 结论

本设计将“业务字段中心”从模板字段汇总工具升级为“字段主数据管理台”，采用“主数据唯一主源 + 模板字段引用绑定 + 兼容回退”的路线，在不破坏现有模板能力的前提下，为业务元数据治理、模板引用追踪和后续规则/字典扩展建立稳定边界。

这是一条可以用单个实现计划覆盖、且能与当前仓库分层方向保持一致的演进路径。
